# Knowledge Repository

Durable knowledge, validated approaches, and architectural decisions — organized for execution, not just reference.

This folder is **shared between two ARIA plugins**:

- **aria-cowork** (running in Claude Cowork) — captures and retrieves knowledge during conversational work
- **aria-knowledge** (running in Claude Code) — captures and retrieves knowledge during engineering work

Both plugins read and write the same `aria-config.md` and operate on the same folder structure. Anything you capture in one mode is available in the other.

See [OVERVIEW.md](OVERVIEW.md) for the design philosophy and how the system works. See [LOCAL.md](LOCAL.md) for project-specific conventions and format templates.

## Structure

```
knowledge/
├── README.md
├── OVERVIEW.md              # Design philosophy and how the system works
├── LOCAL.md                 # Project-specific guide (not managed by plugin)
├── aria-config.md           # Canonical config — shared by aria-cowork and aria-knowledge
├── index.md                 # Tag-based knowledge index (generated by /index)
├── intake/                  # Unprocessed input — backlogs and staging
│   ├── insights-backlog.md
│   ├── decisions-backlog.md
│   ├── extraction-backlog.md
│   ├── rules-backlog.md
│   ├── notes/               # Your own notes and rough drafts
│   ├── attachments/         # Files from elsewhere (screenshots, PDFs)
│   └── clippings/           # Saved external content (URLs, snippets)
├── logs/                    # Operational logs — audit dates and results
│   ├── knowledge-audit-log.md
│   └── config-audit-log.md
├── rules/                   # Living rules and principles
│   ├── working-rules.md
│   ├── user-rules.md
│   ├── change-decision-framework.md
│   └── enforcement-mechanisms.md
├── approaches/              # Validated methodologies and patterns
├── decisions/               # Cross-project architectural decisions (ADR format)
├── guides/                  # Operational knowledge — how things work in your environment
├── references/              # External resources, evaluations, research
└── archive/                 # Retired or superseded content
```

**Code-only additions** (created by aria-knowledge in Code, not aria-cowork): `distill/`, `stitch/`, `intake/pre-compact-captures/`, `intake/ideas/`, `projects/{tag}/`. These are git/repo/hook-bound features that don't apply in Cowork. See [aria-cowork ADR-005](https://github.com/mikeprasad/aria-cowork) for the rationale.

## Folder taxonomy

| Folder | Holds | Think of it as |
|--------|-------|----------------|
| `rules/` | Principles, constraints, policies | "We must / must not" |
| `approaches/` | Validated methodologies, patterns | "How we do X (proven)" |
| `decisions/` | ADRs, architectural choices | "We chose X because Y" |
| `guides/` | Operational knowledge, how things work | "Here's how X works" |
| `references/` | External research, evaluations, bookmarks | "What others say about X" |
| `archive/` | Retired content with pointers | "What we used to do" |
| `intake/` | Staging — backlogs, notes, clippings | "Not yet categorized" |
| `logs/` | Audit timestamps and results | System metadata |

## Conventions

- **File naming:** kebab-case (`api-design-patterns.md`)
- **Dates:** Include `**Last updated:** YYYY-MM-DD` near the top of each file
- **Decision files:** Cross-project ADRs in `decisions/` use `YYYY-NNN-` prefix (e.g., `2026-001-auth-strategy.md`) — self-dating and resetting per year
- **Don't delete — archive:** Move deprecated content to `archive/` with pointers
- **Tag your files:** Include `tags: [tag1, tag2]` in YAML frontmatter for all promoted files
- **`index.md` is generated:** Rebuilt by `/index` and `/audit-knowledge`. Never hand-edit it.

## Frontmatter Schema

Knowledge files use YAML frontmatter between `---` delimiters at the top of each file. The schema is **shared across both ARIA plugins** (aria-cowork and aria-knowledge produce schema-identical files).

### Required fields

```yaml
---
name: <slug>                        # Stable identifier; usually matches filename without .md
description: <one-line summary>     # Human-readable hook
type: <category>                    # rules | approach | decision | guide | reference | intake-* | archive-*
---
```

### Optional discovery fields

These fields shape how `/context`, `/ask`, and `/index` discover and surface the file. Together, the **tag** + **alias** + **semantic-hints** trio forms a layered query-matching strategy with increasing precision: hints (broadest, free-form), aliases (redirects, opinionated), tags (narrowest, canonical).

```yaml
---
name: stripe-webhook-retry-pattern
description: Handling failed webhook deliveries with exponential backoff
type: approach
tags: [stripe, webhooks, retry, idempotency]    # Canonical, controlled vocabulary
semantic-hints:                                  # Free-form phrases for natural-language queries
  - "stakeholder framing for new initiative"
  - "exec summary template"
  - "decision options weighing"
---
```

#### `tags:` — canonical vocabulary

Curated list of tags from the canonical set defined in `index.md` under `## Known Tags`, plus freeform tags. `/context <tag>` matches files with that tag exactly. See [LOCAL.md](LOCAL.md) for the tag convention and the initial known-tag set.

#### `semantic-hints:` — free-form discovery phrases (v0.3.0 / aria-knowledge v2.16.0+)

Optional YAML list of free-form descriptive phrases that match query tokens via substring (case-insensitive, hyphen-normalized).

**When to use:** for knowledge files where the canonical title doesn't capture situational utility — meeting prep templates, doc-anchored captures, decision-framing artifacts. Hints are how a future query reaches the file when the user remembers "something about X" but not the exact tag.

**Matching semantics:**
- **Case-insensitive:** `"Stripe Webhook"` matches `"stripe webhook"` matches `"STRIPE WEBHOOK"`
- **Hyphen-normalized:** `"exponential-backoff"` matches `"exponential backoff"`
- **Whole-query substring:** the query string must be contained in a hint phrase. Query `"webhook failure"` matches hint `"stripe webhook failure"` (query contained in hint). Query `"stripe webhook failure with retry"` does NOT match hint `"stripe webhook failure"` (query is longer).

**Indexing:** `/index` parses `semantic-hints:` from every knowledge file and emits a `## Semantic Hints Index` section in `index.md`. Each phrase becomes an index entry with the list of files declaring it.

**Empty-field handling:** missing or empty `semantic-hints:` field means the file is not indexed under any hint; tag-based indexing is unaffected.

#### `aliases.md` — query-token aliases (v0.3.0 / aria-knowledge v2.16.0+)

Aliases live in their own file (`aliases.md`), not in per-file frontmatter. They redirect freeform query tokens to canonical tags before matching. See [aliases.md](aliases.md) for the full convention.

## Working across surfaces

When you capture knowledge in Cowork (e.g., `/aria-cowork:clip`, `/aria-cowork:ask`), the file lands in this folder via Cowork's attached-folder pattern. When you next open Claude Code with aria-knowledge, the same file is right there — no sync, no daemon, no friction. The OS filesystem IS the sync.
